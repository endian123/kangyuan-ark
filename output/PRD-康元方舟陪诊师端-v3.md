## §1 需求背景

### 1.1 业务背景

康元方舟陪诊平台是一款专注于医疗陪诊服务的O2O平台，连接有陪诊需求的用户（C端）与专业陪诊服务人员（陪同端）。陪同端小程序作为平台的核心服务执行端，承担着登录注册、接单、服务执行、服务记录、个人信息管理等关键职能。本PRD文档针对陪同端小程序的「登录页」「服务页面」与「我的页面」进行详细功能定义与交互说明。

### 1.2 产品目标

- 为陪诊师提供便捷的登录注册入口，支持微信一键授权与手机号登录双通道
- 为陪诊师提供便捷的订单接收与服务执行工具，提升服务效率与质量
- 建立标准化的服务流程（开始服务→服务中→服务完成），确保服务可追溯
- 支持陪诊师入驻申请、资质管理、服务时间配置等自助化操作
- 通过服务数据统计与评价反馈，激励陪诊师提升服务水平
- 保障服务过程数据完整（定位、照片、凭证、短信通知），降低平台运营风险

### 1.3 术语定义

| **术语** | **定义** |
| ------- | ------ |
| 陪诊师 | 通过平台审核，具备专业陪诊服务资质的人员 |
| 就诊人 | 接受陪诊服务的患者或客户 |
| 就诊凭证 | 医院挂号单、缴费单、检查报告等证明服务真实性的材料 |
| OCR | 光学字符识别技术，用于自动识别身份证照片中的文字信息 |
| 电子合同 | 平台与陪诊师之间通过线上签署的具有法律效力的入驻协议 |
| 服务工单 | 记录一次完整陪诊服务过程的订单记录，包含服务时间、地点、过程描述、凭证等 |
| UnionID | 微信用户在开放平台下的唯一标识，用于跨应用识别同一用户 |
| OpenID | 微信用户在当前小程序下的唯一标识 |

#### F1 微信一键授权登录

**用户故事**

> 作为未注册游客，我希望通过微信一键授权登录，以便快速完成注册进入平台。

**验收标准（AC）**

- AC1：页面展示绿色渐变背景，含LOGO、产品名称「康元方舟」、副标题「专业陪诊服务平台」
- AC2：页面底部展示「微信一键授权登录」按钮（绿色圆角大按钮，含微信图标+文字）
- AC3：点击登录按钮前，必须先勾选《用户协议》与《隐私政策》复选框，否则提示「请先同意《用户协议》与《隐私政策》」
- AC4：点击登录按钮后，调用 wx.login() 获取临时登录凭证 code，调用 wx.getUserProfile() 获取用户头像、昵称
- AC5：将 code 发送至后端 /api/auth/wechat-login，后端用 code 换取 OpenID + UnionID + SessionKey
- AC6：后端查询数据库是否存在该 UnionID/OpenID，存在则生成 Token 返回用户信息，不存在则创建新用户并生成 Token
- AC7：前端接收 Token 保存至本地存储，根据用户状态跳转：新用户/未入驻→跳转「我的」页面引导入驻，已入驻→跳转「服务」页面首页

**交互四态**

| **状态** | **说明** |
| ------- | ------ |
| 默认态 | 展示绿色渐变背景、LOGO、产品名称、副标题、两个登录按钮、协议勾选框（默认未选中） |
| 操作态 | 点击登录按钮时按钮短暂高亮，触发微信授权流程 |
| 加载态 | 授权过程中按钮显示加载状态，禁止重复点击 |
| 反馈态 | 授权成功→跳转对应页面；授权失败→提示「微信授权失败，请重试」可重新点击 |

**关键流程**

```
用户点击「微信一键授权登录」按钮
    ↓
【前置校验】是否已勾选协议？
    ├─ 否 → 提示「请先同意《用户协议》与《隐私政策》」
    └─ 是 → 继续
        ↓
调用 wx.login() 获取临时登录凭证 code
        ↓
调用 wx.getUserProfile() 获取用户头像、昵称
        ↓
将 code 发送至后端 /api/auth/wechat-login
        ↓
后端处理：
    1. 用 code 换取 OpenID + UnionID + SessionKey
    2. 查询数据库是否存在该 UnionID/OpenID
        ├─ 存在 → 生成 Token，返回用户信息
        └─ 不存在 → 创建新用户，生成 Token，返回用户信息
        ↓
前端接收 Token，保存至本地存储
        ↓
【判断用户状态】
    ├─ 新用户/未入驻 → 跳转「我的」页面，引导入驻
    └─ 已入驻 → 跳转「服务」页面首页
```

**数据边界**

| **接口** | **方法** | **说明** |
| ------- | ------ | ------ |
| /api/auth/wechat-login | POST | 微信登录，参数：code, userInfo；返回：token, userId, status（未入驻/审核中/已入驻）, name, phone, avatar |

**埋点**

| **事件** | **触发时机** | **参数** |
| ------- | ---------- | ------ |
| login_wechat_click | 点击微信登录按钮 | — |
| login_wechat_agree | 勾选协议 | — |
| login_wechat_success | 微信登录成功 | userId, status |
| login_wechat_fail | 微信登录失败 | errorCode |

**功能级预期收益**

登录成功率 ≥ 95%，新用户注册流程耗时 ≤ 10秒。

#### F2 手机号登录

**用户故事**

> 作为未注册游客，我希望通过手机号+验证码登录，以便在无法使用微信授权时也能登录。

**验收标准（AC）**

- AC1：点击「手机号登录」按钮，跳转手机号登录页面，页面含返回按钮、标题、手机号输入框、验证码输入框、获取验证码按钮、登录按钮
- AC2：手机号输入框前缀+86，校验规则为11位、1[3-9]开头
- AC3：验证码为6位数字，短信发送，5分钟有效
- AC4：点击「获取验证码」后校验手机号格式，格式错误提示「请输入正确的手机号」，格式正确则调用后端发送短信验证码
- AC5：获取验证码后按钮变为60秒倒计时状态，倒计时期间不可重复点击
- AC6：同一手机号24小时内最多发送10条验证码
- AC7：输入验证码后点击「登录」，后端校验验证码是否正确且未过期，已注册→生成Token返回用户信息，未注册→创建新用户生成Token返回用户信息
- AC8：页面底部提示「未注册手机号将自动创建账号」
- AC9：短信模板为「【康元方舟】您的验证码是{123456}，5分钟内有效，请勿泄露。」

**交互四态**

| **状态** | **说明** |
| ------- | ------ |
| 默认态 | 展示手机号输入框、验证码输入框（含获取验证码按钮）、登录按钮、底部提示文字 |
| 操作态 | 输入手机号→点击获取验证码→输入验证码→点击登录 |
| 加载态 | 获取验证码后按钮变为60秒倒计时；登录按钮显示加载状态 |
| 反馈态 | 手机号格式错误→提示并不清空；验证码错误→提示「验证码错误，请重新输入」；验证码过期→提示「验证码已过期，请重新获取」；登录成功→跳转对应页面 |

**关键流程**

```
用户点击「手机号登录」按钮
    ↓
【前置校验】是否已勾选协议？
    ├─ 否 → 提示「请先同意《用户协议》与《隐私政策》」
    └─ 是 → 跳转手机号登录页面
        ↓
输入手机号 → 点击「获取验证码」
    ↓
【校验】手机号格式是否正确？
    ├─ 否 → 提示「请输入正确的手机号」
    └─ 是 → 调用后端发送短信验证码，按钮变为60秒倒计时
        ↓
输入验证码 → 点击「登录」
    ↓
后端处理：
    1. 校验验证码是否正确且未过期（5分钟有效）
    2. 查询手机号是否已注册
        ├─ 已注册 → 生成 Token，返回用户信息
        └─ 未注册 → 创建新用户，生成 Token，返回用户信息
        ↓
前端保存 Token，跳转对应页面
```

**数据边界**

| **字段** | **类型** | **必填** | **校验规则** |
| ------- | ------ | ------ | ---------- |
| 手机号 | String | 是 | 11位，1[3-9]开头 |
| 验证码 | String | 是 | 6位数字，5分钟有效 |

**埋点**

| **事件** | **触发时机** | **参数** |
| ------- | ---------- | ------ |
| login_phone_click | 点击手机号登录按钮 | — |
| login_phone_sendcode | 点击获取验证码 | phone |
| login_phone_success | 手机号登录成功 | userId, status |
| login_phone_fail | 手机号登录失败 | errorCode |

**功能级预期收益**

提供备选登录路径，覆盖无法使用微信授权的场景，登录成功率 ≥ 95%。

#### F3 登录状态管理

**用户故事**

> 作为陪诊师，我希望系统能记住我的登录状态，以便下次打开小程序时自动登录。

**验收标准（AC）**

- AC1：小程序启动（onLaunch）时读取本地存储 Token
- AC2：Token不存在时展示登录页
- AC3：Token存在时调用 /api/auth/verify-token 验证，验证通过则获取用户信息跳转服务页
- AC4：验证不通过时尝试刷新Token，刷新成功则跳转服务页，刷新失败则清除Token展示登录页
- AC5：Token有效期为24小时，Token过期时静默刷新Token，失败则跳转登录页
- AC6：Token无效（被篡改或用户被禁用）时清除本地Token，跳转登录页
- AC7：首次登录的新用户展示登录页，登录后引导入驻

**交互四态**

| **状态** | **说明** |
| ------- | ------ |
| 默认态 | 小程序启动，读取本地Token |
| 操作态 | 验证Token有效性，尝试刷新 |
| 加载态 | 验证过程中显示加载动画 |
| 反馈态 | 验证通过→跳转服务页；刷新成功→跳转服务页；刷新失败→清除Token展示登录页 |

**关键流程**

```
小程序启动（onLaunch）
    ↓
读取本地存储 Token
    ↓
Token 存在？
    ├─ 否 → 展示登录页
    └─ 是 → 调用 /api/auth/verify-token 验证
                ↓
            验证通过？
                ├─ 是 → 获取用户信息，跳转服务页
                └─ 否 → 尝试刷新Token
                            ↓
                        刷新成功？
                            ├─ 是 → 跳转服务页
                            └─ 否 → 清除Token，展示登录页
```

**数据边界**

| **接口** | **方法** | **说明** |
| ------- | ------ | ------ |
| /api/auth/verify-token | GET | 验证Token有效性 |
| /api/auth/refresh-token | POST | 刷新Token |

**埋点**

| **事件** | **触发时机** | **参数** |
| ------- | ---------- | ------ |
| auto_login_success | 自动登录成功 | userId |
| auto_login_fail | 自动登录失败 | reason |

**功能级预期收益**

减少用户重复登录操作，提升使用体验，自动登录成功率 ≥ 90%。

```
    ├─ 点击「隐私协议」→ 展示《用户协议》/《隐私政策》全文
    ├─ 点击「入驻电子合同」→ 展示已签署合同，可下载PDF
    ├─ 点击「关于我们」→ 展示平台介绍、版本号、客服联系方式
    ├─ 点击「清除缓存」→ 清除本地缓存 → 提示「缓存已清除」
    └─ 点击「退出登录」
        ↓
    弹出确认弹窗「确定要退出登录吗？」
        ├─ 确认 → 清除本地Token及用户信息缓存 → 跳转登录页
        └─ 取消 → 返回设置页
```

| 登录响应 | ≤ 3秒 | 包含微信授权+后端验证全流程 |

### 7.2 安全需求（参考）

| **项目** | **要求** |
| ------- | ------ |
| 数据传输 | 全部接口采用HTTPS加密传输 |
| 敏感信息 | 身份证、手机号等加密存储（AES-256） |

| 验证码安全 | 6位随机数字，5分钟有效，错误3次后需重新获取 |
| 防刷机制 | 同一IP/设备限制登录频率，防止暴力破解 |

## §8 边界 & 不做的事

### 8.1 本期范围

- 陪同端小程序「登录页」「服务页面」「我的页面」三个核心页面模块
- 微信一键授权登录 + 手机号登录双通道
- 订单全生命周期管理（待服务→服务中→服务完成）
- 陪诊师入驻全流程（注册→填写信息→提交申请→审核→签署合同→入驻）
- 可服务时间设置（日历视图+批量设置）
- 系统设置（隐私协议、电子合同、清除缓存、退出登录）

### 8.2 不做的事

| **不做事项** | **原因** |
| ---------- | ------ |
| 服务计时功能 | 原始PRD已标注删除线不做，不开发实时计时器 |
| 订单分配算法 | 由后端/平台侧处理，陪同端不涉及分配逻辑 |
| 评价回复功能 | 就诊人评价展示为只读，陪诊师不可回复 |
| 消息推送 | 短信通知为主要通知方式，不涉及推送功能 |
| 多语言支持 | 当前版本仅支持中文 |
| 数据导出 | 陪诊师端不支持数据导出，由后台运营系统处理 |
| 费用结算 | 陪诊师端不涉及费用结算，由后台运营系统处理 |
