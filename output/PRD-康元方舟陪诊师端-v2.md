# 1. 需求背景

## 1.1 业务背景

康元方舟陪诊平台是一款专注于医疗陪诊服务的 O2O 平台，连接有陪诊需求的用户（C 端）与专业陪诊服务人员（陪诊师端）。陪诊师端小程序作为平台的核心服务执行端，承担着登录注册、接单、服务执行、服务记录、个人信息管理等关键职能。

中国老龄化加速、异地就医增多、医疗流程复杂（挂号-候诊-就诊-缴费-取药），导致大量患者（尤其是老年人、孕妇、异地患者）对陪诊服务存在强烈需求。然而传统陪诊市场缺乏规范化管理，服务质量参差不齐、服务过程不可追溯、医患纠纷频发。

## 1.2 核心痛点

| 痛点编号 | 痛点描述 | 影响范围 |
| --- | --- | --- |
| P1 | 患者对陪诊服务不信任，缺乏过程可视化能力 | C 端用户体验差，平台复购率低 |
| P2 | 陪诊师服务过程无记录，纠纷无凭证，平台运营风险高 | 平台法务风险、客诉成本高 |
| P3 | 陪诊师接单、服务、记录分散在多个工具中，效率低 | 陪诊师工作体验差，流失率高 |
| P4 | 服务时间无管控，虚假签到、提前结束等违规行为无法监控 | 服务质量无法保障，品牌声誉受损 |
| P5 | 陪诊师入驻流程不标准，资质审核依赖人工，效率低 | 平台运营成本高，入驻周期长 |

# 2. 目标用户 & 场景

## 2.1 用户画像

| 角色编号 | 角色 | 定义 | 核心诉求 | 使用频率 |
| --- | --- | --- | --- | --- |
| U1 | 未注册游客 | 未登录或未注册的用户 | 快速注册/登录，了解平台功能 | 首次使用 |
| U2 | 未入驻陪诊师 | 已注册但未完成入驻审核的用户 | 了解入驻流程，完成信息填写与资质提交，尽快通过审核 | 一次性 |
| U3 | 已入驻陪诊师 | 通过平台审核，具备接单资格的陪诊服务人员 | 高效接单、规范服务、查看评价、管理个人信息 | 每日高频 |

## 2.2 核心使用场景

### 场景 1：陪诊师首次入驻
陪诊师张女士下载小程序后，通过微信一键登录，系统引导她完成入驻流程：填写基本信息（姓名、手机号、身份证等）、上传学历证书和护士执业资格证、签署电子合同。提交后等待平台审核，审核通过后即可接收订单。

### 场景 2：陪诊师接收并执行服务
陪诊师李先生在小程序中看到一条待服务订单，点击查看详情了解就诊人信息和就诊安排。服务当天到达医院附近，通过 GPS 定位校验（1000 米范围内）并拍照确认后，点击「开始服务」。服务过程中记录关键节点（挂号、候诊、就诊、取药），上传就诊凭证。

### 场景 3：陪诊师完成服务并查看总结
陪诊师王女士完成陪诊服务后，点击「服务完成」按钮，系统校验服务时长（如 <15 分钟则二次确认），自动提交服务记录，发送完成短信给就诊人，并展示服务总结页（时长、记录、凭证、评价入口）。

### 场景 4：陪诊师管理个人信息
陪诊师赵先生需要更新可服务时间，进入「我的」页面点击「服务时间」，设置周一至周五上午可服务。同时修改偏好医院列表，新增 2 家常服务的医院。

### 场景 5：陪诊师查看服务数据和评价
陪诊师刘女士在服务完成 Tab 中查看本月接单数、累计服务数、评分等统计数据，点击已完成订单查看历史服务详情和就诊人评价，用于改进服务质量。

# 3. 用户故事

### US-1：快速登录
**As a** 未注册陪诊师，**I want to** 通过微信一键授权快速登录平台，**so that** 我可以零门槛进入平台了解陪诊服务并开始入驻流程。

### US-2：完成入驻
**As a** 未入驻陪诊师，**I want to** 在小程序中一站式完成个人信息填写、证书上传和协议签署，**so that** 我可以尽快通过审核并开始接单赚钱。

### US-3：GPS 签到并开始服务
**As a** 已入驻陪诊师，**I want to** 在到达医院附近时通过 GPS 定位校验和拍照签到开始服务，**so that** 平台能验证我真实到岗，保护我的服务记录。

### US-4：记录服务过程
**As a** 服务中的陪诊师，**I want to** 随时记录服务过程关键节点并上传就诊凭证，**so that** 服务过程有据可查，遇到纠纷时有完整凭证。

### US-5：完成服务并查看总结
**As a** 完成服务的陪诊师，**I want to** 在服务结束后查看完整的服务总结（时长、记录、凭证），**so that** 我能了解本次服务情况并等待就诊人评价。

#### 微信一键授权登录流程

```
用户点击「微信一键授权登录」按钮
        |
【前置校验】是否已勾选协议？
    |-- 否 --> 提示「请先同意《用户协议》与《隐私政策》」
    |-- 是 --> 继续
        |
调用 wx.login() 获取临时登录凭证 code
        |
调用 wx.getUserProfile() 获取用户头像、昵称（可选）
        |
将 code 发送至后端 /api/auth/wechat-login
        |
后端处理：
    1. 用 code 换取 OpenID + UnionID + SessionKey
    2. 查询数据库是否存在该 UnionID/OpenID
        |-- 存在 --> 生成 Token，返回用户信息
        |-- 不存在 --> 创建新用户，生成 Token，返回用户信息
        |
前端接收 Token，保存至本地存储（wx.setStorageSync）
        |
【判断用户状态】
    |-- 新用户/未入驻 --> 跳转「我的」页面，引导入驻
    |-- 已入驻 --> 跳转「服务」页面首页
```

#### 接口说明

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| /api/auth/wechat-login | POST | 微信登录，参数：code, userInfo |
| 返回字段 | - | token, userId, status（未入驻/审核中/已入驻）, name, phone, avatar |

#### 验收标准（AC）

| AC 编号 | 验收条件 |
| --- | --- |
| AC-login-01 | 未勾选协议点击微信登录，弹出提示「请先同意《用户协议》与《隐私政策》」 |
| AC-login-02 | 勾选协议后点击微信登录，成功调用 wx.login() 并获取 code |
| AC-login-03 | code 发送至后端后，返回 Token 并保存至本地存储 |
| AC-login-04 | 新用户登录后跳转「我的」页面并引导入驻 |
| AC-login-05 | 已入驻用户登录后跳转「服务」页面首页 |

#### 验证码获取规则

- 点击「获取验证码」后按钮变为倒计时状态（60 秒）
- 倒计时期间不可重复点击
- **短信限制：同一手机号 24 小时内最多发送 10 条验证码**
- 验证码有效期：5 分钟
- 短信模板：「【康元方舟】您的验证码是{123456}，5 分钟内有效，请勿泄露。」

### F-start：GPS 签到校验（开始服务）

| 属性 | 内容 |
| --- | --- |
| **F-ID** | F-start |
| **功能名称** | GPS 签到校验（开始服务） |
| **优先级** | P0 |
| **用户故事** | US-3：As a 已入驻陪诊师，I want to 在到达医院附近时通过 GPS 定位校验和拍照签到开始服务，so that 平台能验证我真实到岗，保护我的服务记录。 |

#### 功能描述

陪诊师在待服务订单详情页点击「开始服务」按钮后，系统执行 GPS 签到校验：校验当前定位是否在目标医院 1000 米范围内，弹出拍照上传弹窗（非必填但建议上传），确认后订单状态变更为「服务中」，记录服务开始时间。

#### GPS 签到校验规则

| 规则项 | 规则详情 |
| --- | --- |
| 校验半径 | 以医院经纬度为中心，半径 **1000 米**范围内 |
| 定位精度 | 要求 GPS 定位精度 <= 100 米，精度不足时提示「定位精度不足，请移动到开阔地带」 |
| **容错机制** | **连续 3 次定位失败，允许手动输入「我已到达」并上传现场照片作为佐证** |
| 照片上传 | 支持拍摄或从相册选择，单张 <= 5MB，格式 JPG/PNG，非必填但建议上传 |
| 图片数量限制 | **签到照片最多 1 张** |

#### 操作按钮

| 按钮 | 触发条件 | 执行逻辑 | 异常处理 |
| --- | --- | --- | --- |
| 开始服务 | 陪诊师在订单详情页点击 | 1. 获取当前 GPS 定位 2. 校验定位是否在目标医院 1000 米范围内 3. 弹出拍照上传弹窗（可跳过） 4. 确认后订单状态变更为「服务中」 5. 记录服务开始时间 | 定位校验失败：提示「请到达医院附近后再开始服务」 |
| 发送短信 | 点击按钮 | 调用短信模板，向就诊人发送服务提醒短信（含陪诊师姓名、联系方式） | 短信发送失败：提示「短信发送失败，请检查网络或手动联系」 |
| 电话联系 | 点击按钮 | 调起系统拨号界面，拨打就诊人预留手机号 | 无 |

```
    |

              |

              |-- 否 --> 可重试
              |-- 是 --> 弹出「手动签到」入口
                        |
```

| 属性 | 内容 |
| --- | --- |
| **F-ID** | F-endcheck |
| **功能名称** | 15 分钟服务时长校验 |
| **优先级** | P0 |
| **用户故事** | US-5：As a 完成服务的陪诊师，I want to 在服务时长过短时系统提示确认，so that 避免误操作提前结束服务。 |

#### 功能描述

当陪诊师点击「服务完成」按钮时，系统计算从服务开始到当前时刻的时长。如服务时长 < 15 分钟，弹出二次确认弹窗，要求陪诊师确认是否已完成全部服务。

#### 校验流程

```
陪诊师点击「服务完成」按钮
        |
计算服务时长 = 当前时间 - 服务开始时间
        |
【校验】服务时长 >= 15 分钟？
    |-- 是 --> 直接进入服务完成确认流程
    |-- 否 --> 弹出二次确认弹窗
                |
            「服务时长较短（X 分 X 秒），请确认是否已完成全部服务？」
            [取消]  [确认完成]
                |
            |-- 取消 --> 返回服务中页面
            |-- 确认完成 --> 记录服务时长，进入服务完成流程
```

| AC 编号 | 验收条件 |
| --- | --- |
| AC-endcheck-01 | 服务时长 >= 15 分钟时，直接进入服务完成确认流程 |
| AC-endcheck-02 | 服务时长 < 15 分钟时，弹出二次确认弹窗，显示实际时长 |
| AC-endcheck-03 | 点击「取消」返回服务中页面，不改变订单状态 |
| AC-endcheck-04 | 点击「确认完成」继续执行服务完成流程，记录实际时长 |

#### 四态交互

| 状态 | 说明 |
| --- | --- |
| 默认态 | 服务完成按钮可点击 |
| 警告态 | 服务 <15 分钟，弹出确认弹窗（红色警告图标） |
| 取消态 | 返回服务中页面 |


| 服务记录完整率 | 有记录+凭证的完成订单数 / 已完成订单数 x 100% | >= 80% | < 60% |
| 手动签到率 | 手动签到数 / 总签到数 x 100% | <= 15% | > 25% |
| 15 分钟短时服务率 | 时长 <15 分钟的完成订单数 / 已完成订单数 x 100% | <= 5% | > 10% |

# 8. 边界 & 不做的事

## 8.1 明确不做的事

| 编号 | 不做的事 | 原因 |
| --- | --- | --- |
| B-01 | **取消订单功能** | PRD 澄清确认：陪诊师端不提供取消订单入口，取消操作由 C 端或平台后台处理 |
| B-02 | **实时服务计时器** | 原始 PRD 标记为「不做」，V2.0 不纳入。服务时长由后端根据开始/结束时间自动计算 |
| B-03 | **在线支付** | 陪诊师端不涉及费用结算，支付流程在 C 端完成 |
| B-04 | **实时聊天** | V1.0 不做 IM 聊天功能，通过短信和电话进行沟通 |
| B-05 | **抢单/派单机制** | V1.0 不涉及订单分配策略的 UI 交互，分配逻辑由后台完成 |
| B-06 | **导航/地图功能** | V1.0 仅做 GPS 定位校验，不做路线导航或地图展示 |
| B-07 | **陪诊师互评** | V1.0 仅支持就诊人对陪诊师评价，不支持陪诊师反向评价 |

## 附录 E：术语定义

| 术语 | 定义 |
| --- | --- |
| 陪诊师 | 通过平台审核，具备专业陪诊服务资质的人员 |

| 服务工单 | 记录一次完整陪诊服务过程的订单记录，包含服务时间、地点、过程描述、凭证等 |
| UnionID | 微信用户在开放平台下的唯一标识，用于跨应用识别同一用户 |
| OpenID | 微信用户在当前小程序下的唯一标识 |
| GPS 签到 | 陪诊师到达医院后通过 GPS 定位校验确认到岗的机制 |
| 草稿保存 | 服务记录在未提交前自动保存至本地，防止意外丢失 |
